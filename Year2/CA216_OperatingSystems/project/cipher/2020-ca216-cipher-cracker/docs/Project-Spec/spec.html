Project Description

the objective of this assignment is very simple. You will be given files containing messages which have been encrypted using a cipher. You can assume that the message will be encrypted using either a Caesar Cipher or a substitution cipher. Your job is to crack the code, and find out what the contents of the files are.

however as this is a module which introduces things like processes, threads and the like, there's a wrinkle to all this. You not only have to crack the code, but your programs have to do so as quickly as possible. You can use any of the following:

  • Python's multiprocessing module
  • Threading
  • any other ideas you may have

You may not use any external libraries such as PyCipher etc. However, if you want to test your own code against this, it may or may not be helpful to do so. In other words: This has to be entirely your own work. You will need to write a "normal" (single threaded) application first. Time how long it takes to crack the code. then experiment with parallelising your code. Are there any differences? Are there parts of the code which cannot/should not be parallelised? Play around, experiment, record test results (timings etc.) you'll need this data later.

What You are Required to Do

You will be required to submit working code at the end of this project. Your code needs to be well designed and commented. Also, you will need to write a 1000-1200 word report explaining:

  • How and why you chose the method you did to solve this problem
  • How you tested your program to determine which method was faster. In other words: is the single threaded (as in perfectly normal/traditional) implementation faster than one using multiprocessing, threading, or a combination of both.
  • Explain your results. Why do you think you got the answers you did.
  • What would you do differently were you to undertake this exercise again?

Please note that when I am grading your programs, I will not look at lots of different versions. I will look at the version which includes evidence of threads, processes and the other ideas you have used. I need to stress this, as many of you will have lots of different versions of the program.

What Output Should the Program Produce?

As has been mentioned before, your program should take a text file as input. It should analyse that file and produce two files as its output:

  1. the decrypted message. You should use the following as a naming scheme inputfilename-decrypted.txt
  2. a file containing the key. The file should be named: inputfilename-key.txt and should take the form letter = cipher-letter.

See the examples provided for more details on the format of each of these files.

Step 1: Do It Yourself.

Write a program which takes a file of text and encrypts it using a substitution cipher. Or, find some code online which does this. This way, you know both the plaintext, and the resulting cipher text. You can then work backwards to decrypt it. You will have to:

  • find the key
  • use the key to transform the cipher text back to the plain text

You can, if you wish, start to build in your telemetry once you have a reasonably stable solution. In other words: You can test how long it takes to crack the cipher in the situation where you have all of the facts. This way, you can firstly get a solution with which you're happy, and then add in the timings.

Step 2

At the end of week 8, I will give you files in cipher text. You will be given:

  • the cipher text for all files
  • The plain text for all files
  • The key for all files.
  • You can then apply your own programs to these new files. .

Again, do some testing to determine the impact (if any) of parallelising the process of cracking the cipher. Make notes etc.

Step 3: After Submission

After submission, I will run your code on files which you have not been given. The marks will be awarded for how well your programs crack the messages I use. I will use either a message encrypted using a Caesar Cipher, or a substitution cipher to test your code. You must choose which encryption technique you wish me to use. Note that if you choose a Caesar cipher, you will only receive a maximum grade of 50%, as this technique only involves shifting letters up or down the alphabet, whereas the substitution cipher uses a far more complex key. I will look at the key you produce and compare it against the actual key.

Some Pointers

You will need to do some frequency analysis on the text. Remember that in a Caesar cipher each letter is simply shifted up (or down) the alphabet by a fixed amount. So if the shift is +3, then a becomes d, b becomes e etc. In a substitution cipher, this won't be the case. Here, a random letter is assigned as the cipher-equivalent of a character in the plain-text.

You can, however, use probabilities to help you. If, for example, the letter 'w' appears most often in your cipher-text, then you can be fairly certain that it equates to 'e'. Also, look for commonly occurring groups of two letters. There are only a few of these. You can use this sort of information to help you. In the links at the bottom of this document, and also in the "resources" folder in the gitlab repo, there are resources which will help you with this project.

Checking your Solution

You can verify your solution by inspection. There is no need to automatically look up lots of word dictionaries to see if you have cracked the code successfully. The easiest way to do this is to look at the output and the key. Have you matched all letters? Does each letter correspond to one letter in the cipher alphabet? do the words make sense?

Marking guidelines

The following are the criteria by which the project will be graded.

  1. You will receive 1 mark for each letter in the key you crack correctly. If you get them all, this gets rounded up to 30.
  2. You will receive up to 30 marks for the quality of your solution.
  3. You will receive up to 10 marks for the quality of the testing you have used to determine which version of the code is quicker.
  4. You will receive up to 30 marks for the report.
